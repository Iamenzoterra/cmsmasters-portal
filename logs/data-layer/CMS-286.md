## What Was Implemented

Three ADR markdown files were created in `workplan/adr/`: `018-supabase.md` documents the decision to adopt Supabase as the unified backend (Postgres, Auth, Realtime, Storage) with concrete trade-offs around vendor lock-in, RLS complexity, and free-tier limits. `019-orchestrator-role.md` defines the AI orchestrator's responsibilities for task dispatch, agent boundaries, retry/escalation policy, and audit trail conventions. `020-database-schema.md` specifies a normalized five-table schema (users, themes, licenses, tickets, subscriptions) with explicit FK relationships, RLS ownership patterns, and a relationship summary diagram.

## Key Decisions

| Decision | Chosen | Why |
|----------|--------|-----|
| Schema normalization | Normalized with explicit FKs | Enables clean entitlement joins (ADR-5) and consistent RLS ownership via user_id |
| Orchestrator scope | Development-time only | Runtime AI agents (ADR-13) are a separate concern; conflating them would muddle responsibilities |
| Supabase client sharing | Shared `packages/supabase` library in Nx monorepo | Avoids duplicating connection config across five apps (ADR-17) |
