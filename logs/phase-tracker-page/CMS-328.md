## What Was Implemented
Replaced the existing `phases/page.tsx` (which delegated to a `PhaseCard` component) with a fully inlined async Server Component that handles all filtering, phase expansion, and task detail sheet logic. The page reads `phases.json` via `getPhases()`, parses URL search params (Next.js 15 Promise-based `searchParams`), filters tasks by status and search term, renders expandable `<details>` cards for each phase, and shows a slide-in `TaskDetailSheet` overlay when a task row is clicked. All state is encoded in URL params (`statuses`, `search`, `task`) enabling full server-side rendering with no client state.

## Key Decisions
| Decision | Chosen | Why |
|----------|--------|-----|
| State mechanism | URL params via Link/form GET | Server Component — no useState allowed; enables bookmarkable URLs |
| Phase expansion | Native `<details>`/`<summary>` HTML | Zero JS required; browser handles open/close toggle natively |
| Nested ternary | Extracted to `taskListContent` variable | `sonarjs/no-nested-conditional` ESLint rule enforced as error |
| Search form | `<form method="get">` with hidden `statuses` input | Preserves active status filters on search submit without JS |
| Chevron rotation | `group-open:rotate-180` via Tailwind `group` on `<details>` | Standard Tailwind pattern for details open-state styling |
| Slide-in animation | `@starting-style` CSS inside inline `<style>` tag | No Tailwind equivalent; scoped to `.task-detail-sheet` class |
