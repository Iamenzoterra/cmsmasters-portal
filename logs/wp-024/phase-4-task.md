## Context

Phases 1–3 built the data + write-path + render-path. Phase 4 turns on the **browser-side responsive mechanism** that lets variants actually *do something*: leaf slots expose `container-type: inline-size` so block CSS can write `@container slot (max-width: …) { … }` rules. Separately, a new companion token file `tokens.responsive.css` holds hand-maintained `clamp()`-based rhythm values (decoupled from the Figma-synced `tokens.css`).

Hard invariant for this phase (and the WP overall): **every existing block must render visually identical**. `container-type: inline-size` establishes size containment on the block axis only and does NOT affect layout when no `@container` query references it. Verified in Phase 4.8 with a published-theme regenerate + visual diff.

## Task 4.1: Amend `css-generator.ts` — generic `.slot-inner` rule

### What to Build

File: `tools/layout-maker/runtime/lib/css-generator.ts`, at lines 245-252.

**Current (pre-Phase-4 baseline from Phase 0 RECON):**

```ts
  // Generic slot-inner base rule
  out.push('[data-slot] > .slot-inner {')
  out.push('  display: flex;')
  out.push('  flex-direction: column;')
  out.push('  width: 100%;')
  out.push('  flex: 1 0 auto;')
  out.push('}')
  out.push('')
```

**Target:**

```ts
  // Generic slot-inner base rule.
  // container-type: inline-size + container-name: slot expose each leaf
  // slot's inline width to @container rules inside block CSS, enabling
  // per-slot responsive variants without JS (WP-024 / ADR-025).
  out.push('[data-slot] > .slot-inner {')
  out.push('  display: flex;')
  out.push('  flex-direction: column;')
  out.push('  width: 100%;')
  out.push('  flex: 1 0 auto;')
  out.push('  container-type: inline-size;')
  out.push('  container-name: slot;')
  out.push('}')
  out.push('')
```

### Emits (generated CSS)

```css
[data-slot] > .slot-inner {
  display: flex;
  flex-direction: column;
  width: 100%;
  flex: 1 0 auto;
  container-type: inline-size;
  container-name: slot;
}
```

### Domain Rules

- **Container slots are unaffected.** The per-slot loop (line 256) still `continue`s on container slots — they hold `<div data-slot="child">` directly, not `.slot-inner`, so the generic selector `[data-slot] > .slot-inner` naturally doesn't match them.
- **Order of declarations:** put the two new properties AFTER the existing five. Rule becomes 7 lines of declarations — maintains readability.
- **Do NOT touch the per-slot loop (line 256-267)** — per-slot rules don't need these properties (they inherit via specificity since the selector `[data-slot="X"] > .slot-inner` is more specific than `[data-slot] > .slot-inner` and these two new properties aren't on the per-slot list). CSS containment is a property that layers; leaf slots will have both rules applied in cascade. Let the generic rule own the container properties.

## Task 4.2: Contract test in `css-generator.test.ts`

### What to Build

Extend `tools/layout-maker/runtime/lib/css-generator.test.ts` with a new `describe` block. Use the existing `threeColLayout` helper and `tokens` fixture (top of the file).

Add at the end of the file (or near related slot-inner tests if any exist):

```ts
describe('WP-024: slot container-type (ADR-025)', () => {
  it('emits container-type + container-name on the generic .slot-inner rule', () => {
    const css = generateCSS(threeColLayout(), tokens)
    // The generic rule is unscoped: `[data-slot] > .slot-inner { ... }`.
    const generic = extractRule(css, '[data-slot] > .slot-inner')
    expect(generic).toContain('container-type: inline-size')
    expect(generic).toContain('container-name: slot')
    // Sanity: the five pre-existing declarations still present.
    expect(generic).toContain('display: flex')
    expect(generic).toContain('flex-direction: column')
    expect(generic).toContain('width: 100%')
    expect(generic).toContain('flex: 1 0 auto')
  })

  it('does NOT emit container-type on container-slot outer rules', () => {
    // Container slots (with nested-slots) hold <div data-slot="child"> directly,
    // not .slot-inner — so the generic selector doesn't match them, and the
    // per-slot loop `continue`s on them. Assert the container slot's own outer
    // rule has no container-type declaration.
    // Use a layout with a container slot if the existing fixture doesn't have one;
    // otherwise just assert that no rule matching `[data-slot="content"] { ... }`
    // (outer rule, not `> .slot-inner`) contains container-type.
    const css = generateCSS(threeColLayout(), tokens)
    const outerContainerRules = css.match(/\[data-slot="[^"]+"\]\s*\{[^}]*\}/g) ?? []
    for (const rule of outerContainerRules) {
      expect(rule).not.toContain('container-type')
      expect(rule).not.toContain('container-name')
    }
  })
})

// Helper — extract a single CSS rule body for a given selector (first occurrence).
// If the test file already has an equivalent helper, reuse it.
function extractRule(css: string, selector: string): string {
  const idx = css.indexOf(selector)
  if (idx < 0) return ''
  const open = css.indexOf('{', idx)
  const close = css.indexOf('}', open)
  return css.slice(open + 1, close)
}
```

**Before writing `extractRule`, check:** `grep -n "function extract\|function findRule\|function getRule" tools/layout-maker/runtime/lib/css-generator.test.ts` — reuse if one exists.

### Domain Rules

- **Tests must live in the existing file**, not a new one. Manifest already gets `.test.ts` added in Task 4.9.
- Use the existing `threeColLayout()` + `tokens` fixtures. Do NOT invent new fixtures.
- Both tests in one `describe` block, both assert against real generator output.

/** GET /tokens/css — raw tokens.css + tokens.responsive.css for iframe injection */
blocks.get('/tokens/css', (c) => {
  const baseTokensPath = path.resolve(import.meta.dirname, '../../../../packages/ui/src/theme/tokens.css')
  const responsiveTokensPath = path.resolve(import.meta.dirname, '../../../../packages/ui/src/theme/tokens.responsive.css')
  const parts: string[] = []
  if (existsSync(baseTokensPath)) parts.push(readFileSync(baseTokensPath, 'utf-8'))
  if (existsSync(responsiveTokensPath)) parts.push(readFileSync(responsiveTokensPath, 'utf-8'))
  if (parts.length === 0) {
    c.header('Content-Type', 'text/css')
    return c.body('/* tokens.css + tokens.responsive.css not found */')
  }
  c.header('Content-Type', 'text/css')
  return c.body(parts.join('\n'))
})
