### Slot-wrapper verdict

Grep evidence:
- `apps/portal/lib/hooks.ts:234` → `output += \`<div data-block-shell="${slug}">...\``  (portal production)
- `tools/layout-maker/src/components/Canvas.tsx:54` → `<div data-block-shell="${block.slug}">`  (LM iframe)
- `packages/block-forge-core/src/compose/render-preview.ts:16-17` → emits `<div data-block-shell="{slug}">{html}</div>`  (WP-025)

**All three agree on `data-block-shell="{slug}"`** — this is the block wrapper across portal + LM + block-forge-core. Plan line 254's mention of `.slot-inner` is **not** a `data-block-shell` replacement; it's the CONTAINMENT parent that must sit around `data-block-shell` to make `@container slot (max-width: Npx)` rules (emitted by WP-025 heuristics) evaluate inside the preview iframe.

**Portal production hierarchy (theme-page output):**
```
<div data-slot="X" class="…">
  <div class="slot-inner">                     ← container-type: inline-size; container-name: slot
    <div data-block-shell="slug">…</div>       ← data-block-shell wraps the block html
  </div>
</div>
```
`[data-slot] > .slot-inner { container-type: inline-size; container-name: slot; }` is emitted by `tools/layout-maker/runtime/lib/css-generator.ts:254-255` (per the existing `infra-tooling/SKILL.md` invariant). Portal honors that wrapper.

**LM does NOT put `container-type` on its iframe wrapper** — LM's legacy blocks author with `@media` queries, not `@container slot` queries. Block-forge's scope is the opposite: WP-025 heuristics emit `@container slot (max-width: Npx)` chunks. So block-forge's preview iframe MUST provide that containment context somewhere, OR none of the accepted suggestions will ever resolve in the preview.

**Plan line 254 interpretation verified:** block-forge's `composeSrcDoc` needs to
- call `renderForPreview(block)` → get `{ html: '<div data-block-shell="slug">…</div>', css }`
- wrap that html in an additional `<div class="slot-inner">…</div>`
- inject `.slot-inner { container-type: inline-size; container-name: slot; }` into `@layer shared` (alongside `portal-blocks.css`)

This is a **deliberate divergence from LM** (LM doesn't wrap) and a **deliberate alignment with portal's theme-page hierarchy**. No plan correction on this point; Phase 2 prompt should cite this paragraph so the wrapping is not accidentally omitted.

## 0.7 Save Safety Contract (carry-over f)

Six rules pinned as the spec for Phase 1 `file-io.ts` + Phase 4 Save path. **No implementation yet — this is document-only.**

1. **Read:** `readBlock(path)` rejects if `html` or `slug` is missing/empty — throws a typed error (class / tagged union) whose message cites the offending file path AND the missing field name, so the author can fix the source JSON without guessing.
2. **Write scope:** `writeBlock(path, block)` only writes back to the exact file path handed in at open time. No recursive writes, no new-file creation under `content/db/blocks/`, no writes anywhere outside the opened file. Block-craft remains the sole entrypoint for creating new block JSONs.
3. **Backup policy (first-save-per-session):** the very first `writeBlock` call in a session writes `<sourcePath>.bak` with the pre-save file contents (bytes, not re-serialized). Idempotent: subsequent saves in the same session do NOT re-write the backup. **Session boundary =** fresh dev-server start OR picker-switch to a different block AND back. This is the pinned default per Brain's Phase 0 framing; LM currently has no equivalent policy, so there's no "author habit" to surprise.
4. **Dirty guards:** both (a) picker-switch to a different block and (b) browser `beforeunload` emit a warning banner / native prompt if any suggestion in the current block has an accept/reject state that hasn't been saved. Copy: "Unsaved suggestion state — save or discard before switching?"
5. **No-op Save:** if the current block has zero pending suggestions (nothing accepted since last save), the Save button is disabled; no disk write, no `.bak` write, no re-analysis.
6. **Delete:** block-forge never deletes files under `content/db/blocks/*.json` nor their `*.bak` backups. File removal is a manual author action via the OS/editor or a future dedicated tool; it is out of scope for block-forge MVP and WP-028.
